## 1. Executive Summary

在完全相同的底层模型（Claude Sonnet 4.5）下，我们对比了三个 AI 编程 Agent 框架的综合能力：

```
┌────────────────────────────────────────────────────────────┐
│                    最 终 排 名                              │
├──────┬──────────────┬────────┬─────────────────────────────┤
│  #1  │ Hermes Agent │  7.84  │ ⭐ 零重试 · 自主性最强       │
│  #2  │ Claude Code  │  7.71  │ ⭐ 报告质量高 · 交互最好     │
│  #3  │ OpenClaw     │  5.95  │ ⭐ 代码速度快 · Cat4 全失败  │
└──────┴──────────────┴────────┴─────────────────────────────┘
```

**核心结论**: Agent 框架的编排能力、工具集成和错误恢复策略对最终表现的影响**远大于底层模型本身**——同一模型下，最高分与最低分差距达 1.89 分（31.7%）。

## 7. Hermes Agent 的自进化机制分析

### 7.1 机制全貌

Hermes Agent 以 "the agent that grows with you" 为核心理念，实现了一套**五层闭环自进化系统**：

```
┌───────────────────────────────────────────────────────────┐
│                  Hermes 自进化架构                         │
│                                                           │
│   ┌──────────────┐    ┌──────────────┐                   │
│   │  MEMORY.md   │◄───│   Memory     │ ① 持久化记忆     │
│   │  USER.md     │    │   Manager    │   (~2,200 chars)  │
│   └──────┬───────┘    └──────┬───────┘                   │
│          │                   │                            │
│          ▼                   ▼                            │
│   ┌──────────────┐    ┌──────────────┐                   │
│   │  Skills/     │◄───│   Skill      │ ② 自创建技能     │
│   │  SKILL.md    │    │   Manager    │   (agentskills.io)│
│   └──────┬───────┘    └──────┬───────┘                   │
│          │                   │                            │
│          ▼                   ▼                            │
│   ┌──────────────┐    ┌──────────────┐                   │
│   │  Nudge       │───▶│  Background  │ ③ 周期性检查     │
│   │  Timers      │    │  Review      │   (每10轮)       │
│   │  (10 turns)  │    │  (AIAgent)   │                   │
│   └──────────────┘    └──────────────┘                   │
│                                                           │
│   ┌──────────────┐    ┌──────────────┐                   │
│   │  Session     │◄───│  SQLite      │ ④ 跨会话检索     │
│   │  Search      │    │  FTS5        │   (全文搜索)     │
│   └──────────────┘    └──────────────┘                   │
│                                                           │
│   ┌──────────────┐                                       │
│   │  External    │ ⑤ 外部记忆提供者 (Honcho 等 8 个)    │
│   │  Memory      │                                       │
│   │  Providers   │                                       │
│   └──────────────┘                                       │
└───────────────────────────────────────────────────────────┘
```

各层功能：

| 层 | 名称 | 机制 | 存储 |
|----|------|------|------|
| ① | 持久化记忆 | `memory` 工具写入 MEMORY.md / USER.md | 文件 (~2.2KB) |
| ② | 技能自创建 | `skill_manage` 工具 create/patch/edit/delete | SKILL.md 文件 |
| ③ | 周期性检查 | 每 10 轮自动 spawn 后台 AIAgent 审查对话 | 回调 ①② |
| ④ | 跨会话检索 | `session_search` 工具查询历史对话 | SQLite FTS5 |
| ⑤ | 外部记忆 | Honcho / Mem0 等 8 个 Provider 插件 | 云端 / 本地 |

### 7.2 本次评估中是否被触发？

**结论：未被触发。**

| 检查项 | 结果 | 原因 |
|--------|------|------|
| memory 工具调用 | 未发现 | 单次查询模式，无用户偏好纠正 |
| skill_manage 调用 | 未发现 | 任务独立，无重复模式可学习 |
| 周期性 nudge | 可能未达阈值 | 多数任务 < 10 轮即完成 |
| session_search | 未发现 | 首次运行，无历史会话可查 |
| Honcho / 外部 Provider | 未配置 | 评估环境未设置 Honcho |

**原因分析**：自进化机制设计用于**多会话、长期使用**场景——用户反复纠正 Agent → 记忆保存 → 下次不再犯错；Agent 发现复杂任务解法 → 创建 Skill → 下次直接使用。但本次评估是**单次查询**（`hermes chat -q "..." -Q`），每个任务独立启动一个会话，没有：
- 用户纠正或偏好表达
- 跨任务知识复用机会
- 重复犯错后的学习循环

### 7.3 如果启用自进化会怎样？

假设在交互式长期使用场景下：
1. **Cat1 代码项目**: 完成后 Hermes 会自动创建 "Python CLI 项目最佳实践" 技能，下次类似任务直接调用
2. **Cat2 浏览器**: 遇到 Cloudflare 后 → 记忆 "openai.com/blog 需要特殊处理" → 下次直接绕行
3. **Cat3 监控**: worldtime 连接重置 → 记忆 "worldtimeapi.org 不稳定，备选 timeapi.io"
4. **Cat4 交互**: PR 失败 → 分析原因 → 创建 "开源贡献 PR" 技能改进流程

> ⚠ **重要**: 自进化机制是 Hermes 的核心架构优势，但在标准化评测（一次性、无上下文）中无法体现。如果评测设计为多轮迭代任务，Hermes 的优势将进一步放大。

```
  首次成功率:
  Hermes   ██████████  100%  ← 零重试
  CC       ████████░░   80%  ← 2次 429 + 1次 Cloudflare
  OpenClaw █████░░░░░   50%  ← 4次 context overflow + 2次全失败

  OC 的 context overflow 是其最大系统性弱点
```

### 发现 3: Cat4 是当前 Agent 的天花板

向开源仓库提交 PR 需要: 理解陌生代码 → 识别真实问题 → 修复 → git 操作 → 远程交互。**三个 Agent 无一成功**。这代表了当前 AI Agent 在复杂多步骤真实世界交互方面的能力边界。

### 发现 4: 浏览器操作已相当成熟

Cat2 是三个 Agent 平均分最高的类别（CC 8.50 / OC 7.86 / HA 8.89），表明通过 Playwright/agent-browser 等工具，浏览器自动化已成为 AI Agent 的**基本能力**。

### 发现 5: 错误恢复策略各有千秋

| Agent | 策略 | 优点 | 缺点 |
|-------|------|------|------|
| CC | 创造性绕行（Web Archive） | 解决了别人解决不了的问题 | 耗时极长（1285s） |
| OC | 自动 context compaction | 无需人工干预 | 成功率不稳定（~50%） |
| HA | 完整记录+优雅降级 | 透明度最高 | 未尝试创造性解决方案 |

## 附录 B: 评分维度定义

| 维度 | 权重 | 满分标准 (10) | 及格标准 (6) | 零分 (0) |
|------|------|-------------|-------------|---------|
| **正确性** | 30% | 所有输出正确，所有验证通过 | 核心功能正确，存在边缘问题 | 完全错误或无输出 |
| **完整性** | 25% | 所有子任务完成，额外输出 | 80%+ 子任务完成 | 未尝试或放弃 |
| **质量** | 20% | 代码/报告专业级，格式规范 | 可用但风格一般 | 质量极差 |
| **效率** | 15% | 快速完成，无浪费操作 | 中等耗时，少量重试 | 极慢或大量无效操作 |
| **自主性** | 10% | 零人工干预，自动恢复错误 | 偶尔需重试但能自行恢复 | 需要大量人工干预 |
